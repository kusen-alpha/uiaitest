# UIAI 自动化测试框架 — 可行性分析报告

> **版本**：v1.0 | **日期**：2026-06-12
> **分析依据**：架构设计文档 + 6个参考项目深度分析 + ck目录源码验证 + 2026年大厂真实落地方案

## 1. 项目概述

### 1.1 项目定位

**UIAI** 是一个 AI 驱动的 UI 自动化测试框架，核心理念是：

```
用户描述目标 → Agent自主完成测试 → 经验自动沉淀 → 后续执行加速
```

**四大核心需求**：
1. 自然语言用例驱动执行（主流程）
2. AI 执行自动化测试（Agent自主操控浏览器）
3. 全视觉模式支持（纯视觉驱动，不依赖DOM）
4. 代码录制与回放（执行过程同步记录可复现代码）

### 1.2 架构核心组成

| 组成部分 | 说明 | 参考来源 |
|---------|------|---------|
| **智能自主Agent循环** | 感知-决策-执行闭环 | browser-use Agent Loop |
| **Playwright执行层** | Web端浏览器自动化底座 | playwright-python |
| **多模型意图路由** | locate→VL, plan→LLM, extract→轻量 | Midscene.js Intent Routing |
| **三层缓存系统** | Plan/Locate/Feature缓存 | Midscene.js Cache System |
| **知识沉淀机制** | 需求级/产品级/经验级知识库 | QCon小红书分级上下文 |
| **Watchdog守卫系统** | 14种守卫监控浏览器行为 | browser-use Watchdogs |
| **插件化架构** | BaseExecutor抽象层支持多端扩展 | Midscene.js AbstractInterface |

## 2. 技术选型分析

### 2.1 核心技术选型对比

#### 2.1.1 浏览器自动化层

| 方案 | 优势 | 劣势 | UIAI选型 | 选型理由 |
|------|------|------|---------|---------|
| **Playwright** | 跨浏览器、自动等待、网络拦截、Trace调试、Context隔离 | 依赖Node.js驱动进程 | ✅ **首选** | 功能完备、生态成熟、aria_snapshot(mode="ai")专为AI优化 |
| **Selenium** | 多语言支持、成熟稳定 | 无自动等待、API陈旧、无网络拦截 | ❌ 不选 | 功能落后，2026年大厂已全面迁移Playwright |
| **CDP直连** | 底层控制、性能最优 | 仅Chromium、开发复杂 | ⚠️ 备选 | browser-use采用，但Playwright封装更完善 |
| **Puppeteer** | Google官方、CDP原生 | 仅Chromium、API不如Playwright完善 | ❌ 不选 | 功能受限 |

**结论**：Playwright是2026年Web自动化标准底座，大厂（字节、阿里、腾讯、美团）已统一迁移。

#### 2.1.2 AI Agent调度层

| 方案 | 优势 | 劣势 | UIAI选型 | 选型理由 |
|------|------|------|---------|---------|
| **自研代码Agent** | 全流程可控、可观测、可定制、内网推理 | 开发成本高 | ✅ **首选** | 2026年大厂生产标准方案，合规+稳定双保障 |
| **Claude Code + MCP** | 快速集成、零开发成本 | 黑盒调度、数据出境、不可定制 | ❌ 仅本地开发 | 现状方案明确：禁止生产环境使用 |
| **LangGraph** | Agent编排框架成熟 | 需二次深度改造才能上生产 | ⚠️ 参考架构 | 腾讯WeTest基于LangGraph深度改造 |
| **browser-use Agent** | 开源完整实现、MIT许可 | Python生态、依赖CDP | ✅ **参考实现** | Agent循环、Watchdog、LoopDetector可直接借鉴 |

**结论**：自研代码Agent是唯一生产可行方案，MCP仅限本地开发辅助。

#### 2.1.3 视觉定位层

| 方案 | 优势 | 劣势 | UIAI选型 | 选型理由 |
|------|------|------|---------|---------|
| **纯视觉定位(VL模型)** | 跨平台统一、不依赖DOM | Token成本高、精度有限 | ✅ **视觉模式主选** | Midscene.js验证可行，Canvas/WebGL场景必需 |
| **Accessibility Snapshot** | Token高效、结构化、确定性 | 无DOM场景不可用 | ✅ **DOM模式主选** | Playwright MCP核心创新，LLM友好 |
| **DOM序列化** | 信息完整、元素索引化 | Token消耗大 | ⚠️ 备选 | browser-use方案，复杂页面信息过载 |
| **OCR+图像特征** | 低成本兜底 | 精度不稳定 | ✅ **L3自愈层** | 四层降级策略的视觉兜底方案 |

**结论**：混合定位策略（DOM优先+视觉降级）是最佳方案，与架构设计完全一致。

#### 2.1.4 多模态模型层

| 方案 | 优势 | 劣势 | UIAI选型 | 选型理由 |
|------|------|------|---------|---------|
| **私有化国产VL模型** | 数据不出境、合规、低成本推理 | 需部署运维 | ✅ **生产首选** | 字节Seed-VL、阿里Qwen-VL、腾讯混元已验证 |
| **OpenAI GPT-4o** | 能力最强、API便捷 | 数据出境、成本高、合规风险 | ⚠️ 本地开发 | 仅限本地调试，禁止生产 |
| **UI-TARS** | 专为UI定位优化、开源 | 需私有化部署 | ✅ **定位专用** | Midscene.js推荐，7B模型性价比最优 |
| **Ollama本地模型** | 零成本、数据不出境 | 能力有限、需GPU | ⚠️ 备选 | 离线场景兜底 |

**结论**：私有化国产VL模型是唯一合规方案，UI-TARS作为定位专用模型。
