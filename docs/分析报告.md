## 1. 行业现状回顾与核心约束

### 1.1 大厂生产方案定论（2026）

根据现状方案.md，国内互联网大厂已形成统一共识：

| 维度 | 生产标准方案 | 本地辅助方案 |
|------|-------------|-------------|
| 调度主体 | 自研代码 Agent（Orchestrator） | Claude Code CLI 黑盒 Agent |
| 通信方式 | 进程内直调 Playwright SDK | MCP JSON-RPC 跨进程通信 |
| 模型部署 | 国产 VL 私有化内网推理 | 境外 Claude 云端 API |
| 分布式 CI | Docker 浏览器集群 + 多机分发 | 仅 shell 多进程，无法规模化 |
| 自愈策略 | DOM→视觉 OCR 双兜底、弹窗拦截 | Claude 内置固定重试，无法定制 |
| 业务集成 | 对接 RAG、缺陷平台、云真机、CI | 无扩展入口 |

### 1.2 核心约束

1. **合规红线**：业务数据不出境，境外 API 被内网策略拦截
2. **调度可控性**：生产环境需要全链路可观测、可管控
3. **性能成本**：MCP RPC 增加延迟，海量回归 token 成本不可控
4. **业务适配**：需要内部业务 RAG，通用 AI 不懂平台特有流程

### 1.3 架构趋势

- 确定性固定 Playwright 脚本为主（90% 流量）
- AI Agent 做补位自愈、用例生成、异常遍历（10% 流量）
- 纯 AI 自主跑全流程仅用于探索性测试
- 核心主链路确定性脚本兜底

## 2. 开源技术生态全景扫描

### 2.1 Playwright 生态（核心底座）

#### Playwright Test Agents（v1.56+，2025.10 发布）

Playwright 官方推出的三大 AI Agent，已成为 2026 年 AI 测试自动化的标准架构：

| Agent | 职责 | 输入 | 输出 |
|-------|------|------|------|
| **Planner** | 探索应用，生成测试计划 | 自然语言需求 + seed test | Markdown 测试计划 |
| **Generator** | 将计划转为可执行代码 | Markdown 计划 + 页面上下文 | `.spec.ts` 测试文件 |
| **Healer** | 检测失败，自动修复 | 失败的测试执行 Trace | Git diff 修复补丁 |

**初始化方式**：
```bash
npx playwright init-agents --loop=claude   # Claude Code
npx playwright init-agents --loop=vscode   # VS Code
npx playwright init-agents --loop=opencode # OpenCode
```

**Healer Agent 实测数据**（来源：IJESR 2026.01 论文）：
- 假阳性测试失败减少 73%
- 合法故障平均修复时间（MTTR）降低 45%
- 整体测试套件可靠性提升 62%

#### Playwright MCP Server

| 特性 | 说明 |
|------|------|
| 工具数量 | 70+ 工具（导航、点击、输入、表单、标签页、PDF 等） |
| 核心工具 | `browser_snapshot`（辅助功能树快照，200-400 token/次） |
| 模式 | Snapshot 模式（默认，基于辅助功能树）+ Vision 模式（可选，基于截图） |
| 跨浏览器 | Chrome、Firefox、WebKit、Edge |
| 兼容客户端 | VS Code、Cursor、Windsurf、Claude Code、Claude Desktop |
| GitHub Stars | 25,000+ |

**快速配置**：
```json
{

    "playwright": {
      "command": "npx",
      "args": ["@playwright/mcp@latest"]
    }

```

#### Playwright CLI（Agent CLI）

| 维度 | Playwright MCP | Playwright CLI |
|------|---------------|----------------|
| 最佳用途 | 探索性自动化、agentic loops | 编程智能体（Claude Code, Copilot） |
| 工作原理 | LLM 调用 MCP 工具 | 智能体运行 shell 命令 |
| Token 成本 | 较高（工具模式 + 快照） | 较低（约 4x 差距） |
| 默认模式 | 有头模式 (Headed) | 无头模式 (Headless) |
| 设置 | MCP 客户端 JSON 配置 | `npm install -g @playwright/cli` |

#### Playwright Codegen（测试录制生成器）

- 浏览器操作实时录制为测试代码
- 支持 TypeScript、JavaScript、Python、Java、C#
- 自动检测选择器 + 插入断言
- 适合快速生成基础脚本骨架

#### 优势

|------|------|

- 开发本地快速自测

## 4. 推荐方案：混合分层架构

### 4.1 核心理念

**没有"完美"的单一路线，只有"最适合"的分层组合。**

结合现状方案.md 的大厂实践和开源技术生态，推荐采用 **"自研代码 Agent 中控 + 开源技术底座 + MCP 可选接入"** 的混合分层架构：

```
┌─────────────────────────────────────────────────────────────────────┐
│                        用户交互层                                    │
│   CLI 命令行 │ Web 控制台 │ IDE 插件 │ MCP 客户端（可选）           │
├─────────────────────────────────────────────────────────────────────┤
│                     编排调度层 (Orchestrator)                        │
│   任务调度 │ Agent 协作 │ 流程编排 │ 重试策略 │ 资源管理             │
├─────────────────────────────────────────────────────────────────────┤
│                      AI 能力层                                       │
│   用例生成 Agent │ 自愈 Agent │ 视觉比对 Agent │ 探索遍历 Agent     │
├─────────────────────────────────────────────────────────────────────┤
│                     测试框架层                                        │
│   用例管理 │ 断言引擎 │ 数据驱动 │ Page Object │ Hook 机制          │
├─────────────────────────────────────────────────────────────────────┤
│                     执行抽象层                                        │
│   统一 Action 接口 │ 平台路由 │ 会话管理 │ 状态持久化               │
├─────────────────────────────────────────────────────────────────────┤
│                     平台执行层                                        │
│   Playwright (Web) │ Appium (App) │ Maestro (Mobile) │ 视觉引擎    │
├─────────────────────────────────────────────────────────────────────┤
│                     基础设施层                                        │
│   浏览器池 │ 云真机 │ Docker │ CI/CD │ 报告存储 │ 模型推理服务      │
└─────────────────────────────────────────────────────────────────────┘
```

### 4.2 三层运行模式

| 层级 | 占比 | 模式 | 技术 |
|------|------|------|------|
| **L1 确定性脚本层** | 90% | 手写/生成 Playwright 脚本，Pytest/TestRunner 调度 | Playwright SDK 进程内直调 |
| **L2 AI 辅助层** | 10% | 自研 Agent + 内网 VL 模型，自然语言生成/自愈/探索 | 自研代码 Agent + 私有化 VL |
| **L3 本地开发层** | 按需 | Claude Code + Playwright MCP 快速生成基础脚本 | MCP 协议（仅本地） |

```python
        # 2. 自主遍历页面，发现异常
        # 3. 生成探索报告
        pass

    async def heal_failed_test(self, test_id, error_trace):
        """自愈失败测试"""
        # 1. 分析失败原因
        # 2. 调用 Healer Agent 修复
        # 3. 人工审核修复 diff
        pass
```

### 5.2 多 Agent 协作框架

借鉴 Playwright Test Agents 的 Planner/Generator/Healer 三 Agent 模式，扩展为五 Agent 协作：

| Agent | 职责 | 触发条件 | 输出 |
|-------|------|---------|------|
| **Planner Agent** | 分析需求，生成测试计划 | 用户提交测试需求 | Markdown 测试计划 |
| **Generator Agent** | 将计划转为可执行代码 | 测试计划就绪 | `.spec.ts` / `.spec.py` |
| **Executor Agent** | 执行测试，收集结果 | 测试代码就绪 | 执行结果 + Trace |
| **Healer Agent** | 修复失败测试 | 测试执行失败 | 修复 diff（需人工审核） |
| **Explorer Agent** | 自主探索，发现异常 | 定时触发/手动触发 | 异常报告 + 截图 |

**Agent 间通信**：基于文件系统（Markdown/代码文件）+ 消息队列，无共享内存，可随时暂停/恢复。

### 5.3 模型层设计

| 场景 | 推荐模型 | 部署方式 |
|------|---------|---------|
| DOM 分析 + 操作决策 | Qwen2.5-VL-72B / Seed-VL | 内网私有化部署 |
| 视觉比对 + OCR | PaddleOCR + 自微调 UI-VL | 内网私有化部署 |
| 用例生成（自然语言→代码） | DeepSeek-Coder / Qwen-Coder | 内网私有化部署 |
| 本地快速生成（可选） | Claude API / GPT-4o | 外网 API（仅本地开发） |

## 6. 多平台执行层设计

### 6.1 统一执行抽象

```python
from abc import ABC, abstractmethod
from enum import Enum

class Platform(Enum):
    WEB = "web"
    ANDROID = "android"
    IOS = "ios"
    H5 = "h5"

class Action(ABC):
    """统一操作抽象"""
    @abstractmethod
    async def click(self, locator): pass

    @abstractmethod
    async def type(self, locator, text): pass

    @abstractmethod
    async def assert_visible(self, locator): pass

    @abstractmethod
    async def assert_text(self, locator, expected): pass

    @abstractmethod
    async def screenshot(self) -> bytes: pass

    @abstractmethod
    async def get_accessibility_tree(self) -> dict: pass

class PlaywrightExecutor(Action):
    """Web/H5 执行器 - 进程内直调 Playwright SDK"""
    def __init__(self, browser_type="chromium"):
        self.browser = None  # Playwright Browser 实例

    async def click(self, locator):
        # 使用 getByRole / getByTestId 等弹性定位器
        pass

class AppiumExecutor(Action):
    """App 执行器 - Appium WebDriver"""
    def __init__(self, platform: Platform, capabilities: dict):
        self.driver = None  # Appium WebDriver 实例

    async def click(self, locator):
        # Accessibility ID + OCR 双定位
        pass

class MaestroExecutor(Action):
    """移动端轻量执行器 - Maestro YAML 驱动"""
    async def click(self, locator):
        # YAML flow 执行
        pass

class ExecutorFactory:
    """执行器工厂 - 根据平台自动路由"""
    @staticmethod
    def create(platform: Platform, config: dict) -> Action:
        if platform in (Platform.WEB, Platform.H5):
            return PlaywrightExecutor(config.get("browser_type", "chromium"))
        elif platform in (Platform.ANDROID, Platform.IOS):
            if config.get("executor") == "maestro":
                return MaestroExecutor()
            return AppiumExecutor(platform, config)
```

| 平台 | 执行引擎 | 定位策略 | 特色能力 |

## 7. AI 能力层设计

### 7.1 用例生成 Agent

**输入**：自然语言需求 / PRD 文档 / Figma 设计稿 / Jira Issue
**输出**：Markdown 测试计划 → 可执行测试代码

**流程**：

```
自然语言需求
    │
    ▼
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   Planner    │────►│  Generator   │────►│  人工审核     │
│   Agent      │     │   Agent      │     │  Checkpoint  │
└──────────────┘     └──────────────┘     └──────────────┘
     │                      │                     │
     ▼                      ▼                     ▼
  测试计划.md          .spec.ts/.py          确认/修改
  (场景+步骤)         (可执行代码)           后入库
```

**关键设计**：
- Planner 借鉴 Playwright Test Agents 的 Planner，但增加业务 RAG 上下文
- Generator 参考 Playwright Generator，但支持多语言（TS/Python/Java）
- **人工审核 Checkpoint**：生成代码不自动入库，必须经过人工审核（区别于 Playwright Healer 的自动应用）

### 7.2 自愈 Agent

**触发条件**：测试执行失败
**修复策略**：多级降级定位

```
定位失败
    │
    ▼
┌─────────────────────────────────────────────┐
│ Level 1: 同类型选择器替换                     │
│   getByTestId → getByRole → getByLabel      │
│   XPath 等价替换                              │
├─────────────────────────────────────────────┤
│ Level 2: DOM 邻近元素搜索                     │
│   父子兄弟节点遍历                            │
│   文本内容模糊匹配                            │
├─────────────────────────────────────────────┤
│ Level 3: 视觉定位兜底                         │
│   OCR 文字识别定位                            │
│   图像特征匹配（SIFT/SIMPLE）                 │
│   VL 多模态模型理解页面语义                    │
├─────────────────────────────────────────────┤
│ Level 4: AI 代码修复                          │
│   LLM 分析 Trace + DOM，生成修复 diff         │
│   人工审核后应用                              │
└─────────────────────────────────────────────┘
```

**与 Playwright Healer 的关键区别**：
- Playwright Healer 自动应用修复（可能掩盖真实 Bug）
- 本方案：**所有修复必须人工审核**，Healer 只生成 diff 建议

### 12.2 纯视觉 Agent 实现

```python
class VisualAgent:
    """纯视觉 Agent - 不依赖 DOM，仅通过截图和坐标操作"""

    def __init__(self, vl_model, playwright_page):
        self.vl_model = vl_model      # 私有化 VL 多模态模型
        self.page = playwright_page    # Playwright Page 实例

    async def understand_page(self) -> dict:
        """理解当前页面状态"""
        screenshot = await self.page.screenshot()
        return await self.vl_model.analyze(screenshot)

    async def click_element(self, description: str):
        """通过自然语言描述点击元素"""
        screenshot = await self.page.screenshot()
        coords = await self.vl_model.locate_element(screenshot, description)
        await self.page.mouse.click(coords.x, coords.y)

    async def verify_visual(self, description: str) -> bool:
        """视觉语义验证"""
        screenshot = await self.page.screenshot()
        return await self.vl_model.verify(screenshot, description)
```

## 15. 结论

### 15.1 核心结论

1. **没有"完美"的单一路线**。Claude Code CLI + Playwright MCP 适合本地快速验证，自研代码 Agent + 开源底座适合生产规模化。两者不是互斥，而是互补。

2. **生产环境必须自研调度**。合规、可控、可审计、可扩展——这四条红线决定了生产 CI 不能依赖第三方黑盒 Agent。

3. **Playwright 是 Web 执行底座，不是调度层**。调度权必须自研代码接管，Playwright 只负责浏览器驱动。

4. **AI 是补位，不是替代**。90% 核心链路用确定性脚本，10% 用 AI Agent 做自愈/生成/探索。纯 AI 自主跑全流程仅限探索性测试。

5. **MCP 是本地辅助接口，不是生产协议**。MCP 适合 IDE/CLI 轻量场景，不适合企业级分布式调度。

6. **Playwright Test Agents 是最佳参考**。Planner/Generator/Healer 三 Agent 架构已成为 2026 年行业标准，但需增加人工审核 Checkpoint 和业务 RAG 上下文。

### 15.2 推荐技术栈

| 层级 | 技术选型 |
|------|---------|
| 编排调度 | 自研 Python/TS Orchestrator（参考 LangGraph 二次开发） |
| AI Agent | 自研多 Agent 框架（参考 Playwright Test Agents 架构） |
| Web 执行 | Playwright SDK（进程内直调） |
| App 执行 | Appium + Maestro（双引擎） |
| 视觉引擎 | Playwright toHaveScreenshot + 自建 VL 模型 |
| 模型推理 | Qwen-VL / Seed-VL / 混元（内网私有化） |
| 本地辅助 | Claude Code + Playwright MCP（仅本地） |
| 报告 | Allure + Playwright Trace + 自定义 Dashboard |
| CI/CD | Jenkins / GitHub Actions / GitLab CI |
| 容器化 | Docker + K8s（浏览器池 + 分布式调度） |

### 15.3 风险与缓解

| 风险 | 缓解措施 |
|------|---------|
| 自研周期长 | 分阶段实施，Phase 1 即可产出可用框架 |
| VL 模型精度不足 | DOM+视觉双模态融合，DOM 为主视觉兜底 |
| AI 生成代码质量不稳定 | 人工审核 Checkpoint，不自动入库 |
| 自愈误修复掩盖 Bug | 所有修复必须人工审核，Healer 只提建议 |
| 移动端兼容性复杂 | Appium + Maestro 双引擎，按场景选择 |
| Token 成本 | 生产环境全内网推理，边际成本极低 |

---

> **最终建议**：采用"自研代码 Agent 中控 + 开源技术底座 + MCP 可选接入"的混合分层架构。生产环境走自研调度 + 内网模型 + Playwright/Appium 进程内直调；本地开发可选 Claude Code + MCP 快速生成。这不是折中，而是当前技术生态下最务实、最可持续的方案。
