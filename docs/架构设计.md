# UIAI 自动化测试框架 — 架构设计文档

> **版本**：v5.0 | **日期**：2026-06-12

***

## 1. 项目定位

**UIAI** = **智能自主 Agent** + **Playwright 执行层** + **多模型意图路由** + **三层缓存** + **知识沉淀** + **Watchdog守卫** + **插件化架构**

核心理念：**用户描述目标 → Agent自主完成测试 → 经验自动沉淀 → 后续执行加速**

<br />

**四大核心需求**：

1. **自然语言用例驱动执行**（主流程）：测试用例由人工提供（自然语言描述），框架的核心工作是**理解并执行**这些用例，而非生成用例。AI 将自然语言步骤解析为浏览器操作，完成测试执行。
2. **AI 执行自动化测试**：AI Agent 理解自然语言用例后自主操控浏览器完成测试。底层 Web 端基于 Playwright，通过兼容层（BaseExecutor ABC）支持多端扩展，其他端（App 等）暂不实现但接口预留，最大化代码复用。
3. **全视觉模式**：支持纯视觉驱动的测试执行——AI 通过截图 + VL 模型"看"页面，通过坐标/OCR 完成交互，不依赖 DOM 和选择器，模拟人类真实操作方式。
4. **代码录制与回放**：AI 执行过程中同步记录可靠的可复现代码，支持回放和脚本化运行。全视觉模式可跳过代码输出，其他端难实现代码的也不需要，**Web 端优先保证**。

> **核心流程**：`自然语言用例 → PlannerAgent解析 → 执行（DOM/视觉）→ 录制代码 → 报告 → 知识沉淀`

## 3. 核心流程

### 3.1 智能Agent执行循环

> PlannerAgent首先将自然语言用例解析为结构化TestCase，然后进入执行循环。

```
自然语言用例: "登录系统，验证跳转到首页"
    │
    ▼
┌──────────────────────────────────────────────────┐
│              PlannerAgent.parse()                 │
│                                                   │
│   [解析] 自然语言用例 → TestCase                   │
│       ├── 提取测试场景和步骤                        │
│       ├── 识别关键断言点                           │
│       └── 生成PlanItem列表                        │
│                                                   │
└──────────────────────────────────────────────────┘
    │
    ▼
┌──────────────────────────────────────────────────┐
│              PlannerAgent.plan()                  │
│                                                   │
│   [前置] 执行 initial_actions（确定性预步骤）      │
│                                                   │
│   while not done:                                 │
│       │                                           │
│       ├── [感知] 获取页面状态                       │
│       │       DOM / 截图 / URL                     │
│       │       ↓ 查询缓存                           │
│       │       Plan缓存命中？→ 直接执行缓存流程       │
│       │       Locate缓存命中？→ 跳过VL定位调用      │
│       │                                           │
│       ├── [决策] LLM分析下一步                      │
│       │       thinking + action[] + plan_update    │
│       │       ↓ 消息压缩（超阈值自动压缩历史）       │
│       │       ↓ Flash Mode（简单任务跳过冗余字段）   │
│       │                                           │
│       ├── [执行] ExecutorAgent执行动作列表          │
│       │       click / input / navigate             │
│       │       ↓ DOM不可用/视觉场景？→ GeneratorAgent接管 │
│       │       │   └── 两阶段深度定位 + 视觉代码生成       │
│       │       ↓ Watchdog 守卫检查                   │
│       │       ↓ 执行失败？→ HealerAgent四层自愈     │
│       │       ↓ 动作超时？→ ActionTimeout 中断保护  │
│       │       ↓ LLM故障？→ FallbackLLM 自动降级    │
│       │                                           │
│       ├── [后处理] 循环检测 + 目标判断               │
│       │       ↓ Watchdog 事件处理                   │
│       │       ↓ 敏感数据自动脱敏                    │
│       │                                           │
│       ├── [缓存] 成功步骤写入缓存                    │
│       │                                           │
│       └── 目标完成 → break                         │
│                                                   │
│   [评估] JudgeAgent独立评估执行结果                  │
│   [断言] AssertionEngine多层级断言验证              │
│   [沉淀] 成功经验自动写入KnowledgeManager            │
│                                                   │
└──────────────────────────────────────────────────┘
    │
    ▼
录制代码 + 生成报告 + 知识沉淀
```

### 3.2 AgentOutput 数据模型

```python
class AgentOutput:
    thinking: str                      # 内部思考
    evaluation_previous_goal: str      # 评估上一步
    memory: str                        # 跨步记忆
    next_goal: str                     # 下一步目标
    action: list[ActionModel]          # 动作列表（每步可多个）
    is_done: bool                      # 目标是否完成

    # 规划能力（参考browser-use PlanItem）
    current_plan_item: Optional[int]   # 当前执行到哪个计划项
    plan_update: Optional[list[str]]   # 计划更新（允许动态调整计划）

    # 结构化输出
    structured_output: Optional[dict]  # 结构化输出，支持JSON Schema约束
```

### 3.3 TestResultSchema 结构化输出

> v5.0 新增结构化输出能力，支持通过 JSON Schema 约束 Agent 输出格式，确保测试结果可被下游系统自动解析和处理。

```python
class TestResultSchema:
    """测试结果结构化输出Schema"""

    # 基础信息
    test_id: str                       # 测试唯一标识
    test_name: str                     # 测试名称
    status: str                        # 通过/失败/跳过/错误
    duration_ms: int                   # 执行耗时（毫秒）

    # 执行详情
    steps: list[StepResult]            # 每步执行结果
    assertions: list[AssertionResult]  # 断言结果列表
    screenshots: list[str]             # 截图路径列表
    trace_path: Optional[str]          # Trace文件路径

    # AI相关
    llm_calls: int                     # LLM调用次数
    token_usage: TokenUsage            # Token消耗统计
    cache_hits: int                    # 缓存命中次数

    # 自愈信息
    heal_attempts: list[HealRecord]    # 自愈尝试记录

    # 结构化数据提取
    extracted_data: Optional[dict]     # 从页面提取的结构化数据
```

**结构化输出流转**：

```
Agent执行完成
    │
    ├── AgentOutput.structured_output 填充
    │       ↓ JSON Schema 约束格式
    │       ↓ 支持自定义Schema
    │
    ├── TestResultSchema 构建
    │       ↓ 聚合执行/断言/自愈/缓存信息
    │       ↓ 结构化数据提取
    │
    └── 输出到下游
            ├── ReportManager 生成报告
            ├── CI/CD 质量门禁
            ├── 数据分析平台
            └── 知识图谱构建
```

> **设计原则**：结构化输出通过 JSON Schema 约束，确保输出格式一致性和可解析性。支持自定义Schema适配不同业务场景。

### 3.4 Flash Mode

> 参考 browser-use Flash Mode，简单任务跳过冗余输出字段，减少 token 消耗。

| 模式          | 输出字段                                                 | 适用场景          |
| ----------- | ---------------------------------------------------- | ------------- |
| **标准模式**    | thinking + evaluation + memory + next\_goal + action | 复杂多步任务        |
| **Flash模式** | memory + action                                      | 简单单步操作（点击、输入） |

### 3.5 初始动作

> 参考 browser-use `initial_actions`，在LLM循环前执行确定性预步骤，减少token消耗。

```
initial_actions: [
    { navigate: { url: "https://app.example.com/login" } },
    { input: { index: 1, text: "<username>" } },
    { input: { index: 2, text: "<password>" } },
    { click: { index: 3 } }
]
↓ 以上步骤不消耗LLM token，直接执行
↓ 进入LLM循环处理后续非确定性步骤
```


### 3.6 长链路检查点机制

> 长链路测试（如多步表单、跨页流程）成功率随步骤数递减，检查点机制实现分步验证和失败回滚。

`python
class Checkpoint:
    """检查点"""
    step_index: int                    # 步骤序号
    url: str                           # 页面URL
    storage_state: dict                # Cookie/LocalStorage快照
    plan_item_id: int                  # 对应PlanItem
    timestamp: datetime                # 创建时间

class CheckpointManager:
    """检查点管理器"""

    def save(self, step_index: int, context: BrowserContext) -> Checkpoint:
        """保存当前状态为检查点"""

    def restore(self, checkpoint: Checkpoint, context: BrowserContext) -> None:
        """恢复到指定检查点"""

    def get_latest(self) -> Optional[Checkpoint]:
        """获取最近检查点"""

    def cleanup(self, before_step: int) -> None:
        """清理过期检查点"""
`

**检查点与PlanItem状态机联动**：

| PlanItem状态 | 说明 | 检查点操作 |
 | --- | --- | --- |
 | pending | 待执行 | 无 |
 | current | 执行中 | 执行前保存检查点 |
 | done | 已完成 | 保留检查点（可用于回滚） |
 | failed | 失败 | 回滚到最近检查点，重新规划 |
 | retry | 重试中 | 从检查点恢复执行 |
 | blocked | 阻塞 | 等待依赖完成 |
 | skipped | 跳过 | 清理对应检查点 |
 | rollback | 回滚中 | 恢复检查点状态 |

> **设计原则**：检查点仅在关键步骤（页面跳转、表单提交）保存，避免每步保存的性能开销。CrashWatchdog恢复时优先使用最近检查点。

## 4. 三层运行模式

| 层级     | 模式        | 占比     | 说明                                 |
| ------ | --------- | ------ | ---------------------------------- |
| **R1** | 确定性脚本     | 60-90% | 手写/录制的Python代码，直调Playwright        |
| **R2** | 智能Agent辅助 | 10-25% | Agent感知(DOM优先/视觉降级) → LLM决策 → 执行循环 |
| **R3** | 本地开发      | 按需     | MCP Server + Claude Code CLI       |

**占比说明**：

| 层级 | 占比范围 | 推荐场景 |
| --- | --- | --- |
| **R1** | 60-90% | 核心业务链路推荐90%（高稳定性要求），辅助功能推荐60%（允许一定灵活性） |
| **R2** | 10-25% | 复杂场景推荐25%（需AI处理变化性），简单场景推荐10%（确定性操作为主） |
| **R3** | 按需 | 仅本地开发调试使用，不推荐生产环境 |

> **原则**：核心业务链路用R1确定性脚本兜底，AI Agent处理变化性和复杂性，R3仅限本地开发辅助。R2 内部按感知模式细分：DOM感知优先（A11ySnapshot/DOM序列化），视觉感知降级（截图+VL模型），两者不是独立运行模式而是同一层内的感知策略差异。

## 5. Watchdog 守卫系统

> 参考 browser-use 的 14 种 Watchdog 和 QCon 58同城的智能弹窗处理，构建生产级守卫体系。

| Watchdog                  | 守卫对象                       | 行为                                         |
| ------------------------- | -------------------------- | ------------------------------------------ |
| **PopupWatchdog**         | JS弹窗（alert/confirm/prompt） | 自动关闭，将消息注入Agent上下文                         |
| **CaptchaWatchdog**       | CAPTCHA验证码                 | 检测到验证码时暂停，支持2FA/OTP自动处理（pyotp），无法自动处理时通知人工介入                   |
| **CrashWatchdog**         | 浏览器崩溃                      | 自动重启浏览器，恢复到最近检查点（URL+StorageState），从检查点重新执行                           |
| **SecurityWatchdog**      | 域名安全                       | 拦截对非白名单域名的访问                               |
| **NetworkWatchdog**       | 网络异常                       | 检测请求失败/超时，触发重试或降级                          |
| **StorageWatchdog**       | 存储状态                       | 管理Cookie/LocalStorage的保存与恢复                |
| **DownloadWatchdog**      | 文件下载                       | 追踪下载进度，等待下载完成                              |
| **DomWatchdog**           | DOM变更                      | 监控页面结构变化，触发缓存失效                            |
| **VisualAnomalyWatchdog** | UI样式异常                     | 检测空坑(图片空白)、元素堆叠(Toast重叠)、内容异常(NaN/乱码)、白屏报错 |

**Watchdog 工作机制**：

```
Agent执行动作
    │
    ├── EventBus 分发事件
    │
    ├── 各 Watchdog 监听并处理
    │   ├── PopupWatchdog: 拦截弹窗 → 自动关闭 → 注入上下文
    │   ├── CaptchaWatchdog: 检测验证码 → 暂停Agent → 通知人工
    │   ├── CrashWatchdog: 检测崩溃 → 重启浏览器 → 恢复状态
    │   └── SecurityWatchdog: 检测域名 → 拦截非法访问
    │
    └── 处理结果反馈给Agent上下文
```

> **设计原则**：Watchdog 通过事件驱动解耦，不阻塞Agent主循环，可独立添加/移除。

## 6. 安全与可靠性保障

### 6.1 敏感数据保护

> 参考 browser-use `sensitive_data` 机制，LLM只看到键名，实际值在执行时自动替换。

```
配置:
  sensitive_data:
    "app.example.com":
      username: "admin"
      password: "s3cret_token"

LLM看到的: "输入 <username> 和 <password>"
实际执行时: 自动替换为 "admin" 和 "s3cret_token"
历史记录: 自动脱敏，只保留键名

**视觉模式敏感数据保护**：

| 保护层级 | 机制 | 说明 |
 | --- | --- | --- |
 | 截图区域脱敏 | 敏感元素标记+区域模糊 | 标记type=password等敏感元素，截图时自动模糊处理 |
 | OCR检测脱敏 | OCR扫描+模式匹配 | 截图后OCR扫描，检测密钥/令牌模式并脱敏 |
 | 历史截图脱敏 | 存储加密+访问审计 | 历史截图加密存储，访问需审计日志 |
 | 上下文注入脱敏 | 键名替换 | LLM上下文中敏感值替换为<REDACTED>，执行时自动还原 |

> **设计原则**：视觉模式下的敏感数据保护是安全底线，不可因便利性妥协。所有截图在注入LLM前必须经过脱敏检查。

```

## 8. 两阶段深度定位

> 参考 Midscene.js 的深度定位策略，提高复杂页面定位精度，减少token消耗。

```
全页截图
    │
    ▼
┌─────────────────────────────────────┐
│  阶段1: 区域定位（AiLocateSection）   │
│  VL模型识别目标元素所在的大致区域        │
│  输出: 区域矩形 {x, y, w, h}         │
│  裁剪区域时扩展20%边界，确保元素完整可见   │
└─────────────────┬───────────────────┘
                  │
                  ▼
         按区域裁剪截图（含20%扩展边界）
                  │
                  ▼
┌─────────────────────────────────────┐
│  阶段2: 精确定位（AiLocateElement）   │
│  在裁剪后的局部图片中精确定位目标元素     │
│  输出: 元素中心坐标 + 元素矩形          │
└─────────────────────────────────────┘
```

**裁剪区域扩展说明**：

| 扩展策略 | 说明 |
| --- | --- |
| **扩展20%边界** | 区域矩形四边各向外扩展20%，确保目标元素完整可见，避免边缘截断 |
| **边界限制** | 扩展后不超过原图边界，超出部分截断到原图边缘 |
| **收益** | 提高定位精度，减少因元素部分截断导致的定位失败 |

**收益**：

- 减少token消耗（只处理局部图片而非全页截图）
- 提高复杂多元素场景定位精度
- 与Locate缓存联动：缓存命中时直接跳过两阶段定位

### 13.4 InputPrimitives跨平台输入原语

> 跨平台输入原语抽象，新平台只需实现原语接口即可自动获得完整动作空间。

```python
class InputPrimitives(ABC):
    # 跨平台输入原语接口

    @abstractmethod
    def tap(self, element) -> ActionResult: ...

    @abstractmethod
    def type_text(self, element, text: str) -> ActionResult: ...

    @abstractmethod
    def swipe(self, direction: str, distance: int) -> ActionResult: ...

    @abstractmethod
    def scroll(self, direction: str, amount: int) -> ActionResult: ...

    @abstractmethod
    def long_press(self, element, duration: int) -> ActionResult: ...

    @abstractmethod
    def drag(self, source, target) -> ActionResult: ...
```

> **设计原则**：Web平台通过Playwright实现InputPrimitives，App平台通过Appium实现。原子ACTION在InputPrimitives基础上构建，新平台接入只需实现6个原语接口。

### 13.5 核心原子ACTION集

> 参考淘宝闪购的原子ACTION封装和58同城的Skills能力矩阵，定义不可再分的最小操作单元。Skill在原子ACTION基础上组合。

| 原子ACTION            | 说明   | Web映射                             | App映射                    |
| ------------------- | ---- | --------------------------------- | ------------------------ |
| **NAVIGATE**        | 页面导航 | page.goto()                       | driver.start\_activity() |
| **CLICK**           | 点击元素 | element.click()                   | driver.tap()             |
| **TYPE**            | 输入文本 | element.fill()                    | driver.set\_value()      |
| **SELECT**          | 下拉选择 | element.select\_option()          | driver.select()          |
| **SCROLL**          | 滚动   | page.mouse.wheel()                | driver.scroll()          |
| **SWIPE**           | 滑动   | mouse模拟                           | driver.swipe()           |
| **WAIT**            | 等待   | page.wait\_for()                  | WebDriverWait            |
| **SCREENSHOT**      | 截图   | page.screenshot()                 | driver.screenshot()      |
| **ASSERT\_VISIBLE** | 断言可见 | expect(locator).to\_be\_visible() | element.is\_displayed()  |
| **ASSERT\_TEXT**    | 断言文本 | expect(locator).to\_have\_text()  | element.text             |
| **LONG\_PRESS**     | 长按   | mouse模拟                           | driver.long\_press()     |
| **DRAG**            | 拖拽   | page.mouse.drag()                 | driver.drag()            |

> **设计原则**：原子ACTION是L1脚本和L2 Agent的动作空间统一基础，也是代码录制映射规则的标准化依据。Skill在原子ACTION基础上组合为高层业务能力。新平台接入时只需实现原子ACTION到平台API的映射。

## 14. 消息压缩机制

> 参考 browser-use 的消息压缩设计，控制长流程用例的token消耗。

| 参数                       | 默认值   | 说明         |
| ------------------------ | ----- | ---------- |
| compact\_every\_n\_steps | 15    | 每15步压缩一次   |
| trigger\_threshold       | 40000 | 字符数超限触发压缩  |
| keep\_recent\_steps      | 6     | 保留最近6步完整历史 |
| max\_summary\_length     | 6000  | 压缩摘要字符上限   |

**压缩策略**：保留最近N步完整消息 + 将更早的历史压缩为摘要，注入Agent上下文。

> **压缩质量保障**：压缩后摘要需通过信息完整性校验——关键操作（点击、输入、导航）不可丢失，页面状态变更不可遗漏。压缩失败时回退到保留更多历史步数。

## 15. 独立评估机制

> 参考 browser-use 的 Judge LLM 设计，用独立LLM评估Agent执行结果。

```
Agent执行完成
    │
    ▼
┌─────────────────────────────────────┐
│        JudgeAgent.evaluate()        │
│                                     │
│  输入: 任务目标 + 执行轨迹(截图+操作历史)  │
│  输出:                               │

│    failure_reason: 失败原因(如有)       │
│    impossible_task: 任务是否不可完成      │

    │
    ▼
```

## 22. Accessibility Snapshot 感知模式

> v5.0 新增 A11ySnapshot 感知模式，基于 Accessibility Tree 提供结构化页面感知能力。

### 22.1 感知模式引擎

| 感知模式             | 输入                      | Token消耗         | 适用场景                |
| ---------------- | ----------------------- | --------------- | ------------------- |
| **A11ySnapshot** | Accessibility Tree结构化文本 | 200-2000 token（视页面复杂度）   | 标准DOM页面，结构化定位       |
| **DOM序列化**       | DOM树精简序列化               | 500-2000 token  | 需要完整DOM信息的场景        |
| **视觉截图**         | 页面截图 + VL模型             | 1000-3000 token | Canvas/WebGL/复杂视觉场景 |
| **混合模式**         | A11ySnapshot + 截图       | 1200-3400 token | 复杂页面，需双重验证          |

### 22.2 复杂页面感知增强

> 复杂页面（Shadow DOM、跨域iframe、动态加载）需要增强感知能力。

| 增强能力 | 说明 | 实现方式 |
 | --- | --- | --- |
 | Shadow DOM穿透 | 识别Shadow Root内元素 | pierce_shadow=true，递归遍历Shadow Root |
 | 跨域iframe感知 | 获取跨域iframe内容 | CDP协议+权限声明，不可访问时标注 |
 | 遮挡过滤 | 忽略被遮挡的不可交互元素 | z-index+visibility检测 |
 | 截图压缩 | 大页面截图压缩减少token | 自适应压缩比，保留关键区域高清 |
 | 动态等待 | 等待异步内容加载完成 | 网络空闲检测+DOM稳定检测 |

### 22.3 A11ySnapshot 工作流


```
页面加载完成
    │
    ▼
Playwright page.accessibility.snapshot()
    │
    ▼
结构化文本输出（示例）:
  heading "登录" [level=1]
  textbox "用户名" [editable]
  textbox "密码" [editable]
  button "登录" [name="submit"]
  link "忘记密码"
    │
    ▼
注入Agent上下文 → LLM基于结构化信息决策
```

### 22.4 感知降级策略

```
A11ySnapshot（首选，低token，200-2000 token视页面复杂度）
    │ 失败/不完整（如Canvas/WebGL/Shadow DOM内容缺失）
    ▼
DOM序列化（中等token，含Shadow DOM/跨域iframe等完整信息）
    │ 失败/不完整（如无DOM场景）
    ▼
视觉截图（高token，全场景覆盖，Canvas/WebGL兜底）
```

> **说明**：混合模式（A11ySnapshot+截图）是独立可选模式，用于需双重验证的复杂页面，不作为降级链末端。A11ySnapshot失败后不应再回退到包含A11ySnapshot的混合模式。感知模式选择后自动约束意图路由：A11ySnapshot/DOM序列化→LLM定位，视觉截图→VL模型定位。

## 25. Flaky测试治理

> v5.0 新增 Flaky 测试治理系统，自动识别、隔离和处理不稳定测试。

### 25.1 FlakyGovernor 架构

```
FlakyGovernor
  ├── Flaky检测
  │   ├── 历史通过率分析
  │   ├── 执行时间波动分析
  │   └── 失败模式聚类
  │
  ├── Flaky标记
  │   ├── 自动标记（通过率 < 阈值）
  │   ├── 人工标记
  │   └── 标记等级（疑似/确认/严重）
  │
  ├── Flaky隔离
  │   ├── 独立执行环境
  │   ├── 多次重试验证
  │   └── 结果不计入主报告
  │
  └── Flaky修复建议
      ├── 根因分析（时序/依赖/竞态）
      ├── 修复策略推荐
      └── 自动化修复尝试
```

### 25.2 Flaky检测规则

| 指标     | 阈值      | 说明           |
| ------ | ------- | ------------ |
| 历史通过率  | < 80%   | 10次执行中2次以上失败 |
| 执行时间CV | > 0.5   | 执行时间变异系数过大   |
| 失败聚类   | 同模式≥3次  | 相同失败模式重复出现   |
| 间隔失败   | 交替通过/失败 | 通过-失败交替出现    |

### 25.3 Flaky处理策略

```
测试执行失败
    │
    ├── 查询FlakyGovernor
    │   ├── 已标记Flaky？→ 自动重试（最多3次）→ 多数结果为准
    │   └── 未标记？→ 记录失败 → 分析是否Flaky
    │
    ├── Flaky确认流程
    │   ├── 历史数据分析 → 计算通过率
    │   ├── 失败模式聚类 → 识别根因
    │   └── 自动/人工标记 → 隔离处理
    │
    └── 报告输出
        ├── Flaky测试单独标记
        ├── 不影响整体通过率统计
        └── 生成Flaky治理报告
```

### 25.4 Flaky标记复审机制

> Flaky标记需要定期复审，防止标记过时或误标。

| 复审触发 | 条件 | 动作 |
 | --- | --- | --- |
 | 周期复审 | 每月1次 | 重新计算通过率，更新标记等级 |
 | 连续通过 | 连续10次通过 | 自动降级（严重→确认→疑似→移除） |
 | 标记争议 | 人工标记与自动标记不一致 | 触发人工复审 |
 | 环境变更 | 测试环境/配置变更 | 重新评估所有Flaky标记 |

> **设计原则**：Flaky标记不是永久标签，需要持续跟踪和定期复审，确保标记的准确性和时效性。

## 28. 错误处理与恢复架构

> v5.0 新增系统级错误处理与恢复架构，通过熔断器、舱壁隔离和优雅降级保障系统可靠性。

### 28.1 错误分类体系

| 错误类型                    | 说明      | 示例               | 处理策略       |
| ----------------------- | ------- | ---------------- | ---------- |
| **TransientError**      | 短暂可恢复错误 | 网络超时、限流、临时不可用    | 自动重试+指数退避  |
| **PersistentError**     | 持续性错误   | 元素不存在、页面404、断言失败 | 降级处理+错误报告  |
| **InfrastructureError** | 基础设施错误  | 浏览器崩溃、容器OOM、磁盘满  | 资源重建+状态恢复  |
| **BusinessError**       | 业务逻辑错误  | 数据不一致、权限不足、流程异常  | 人工介入+错误上下文 |

### 28.2 熔断器模式

```python
class CircuitBreaker:
    """熔断器：防止级联故障"""

    states: CLOSED → OPEN → HALF_OPEN

    # 参数
    failure_threshold: int = 5       # 连续失败次数阈值
    recovery_timeout: int = 30       # 熔断恢复超时（秒）
    half_open_max_calls: int = 3     # 半开状态最大试探调用数

    # 状态转换
    CLOSED:   正常调用，失败计数 < threshold
    OPEN:     熔断，直接返回错误，不发起调用
    HALF_OPEN: 试探性调用，成功→CLOSED，失败→OPEN
```

**熔断器应用场景**：

| 组件            | 熔断条件     | 熔断行为           |
| ------------- | -------- | -------------- |
| LLM Client    | 连续5次调用失败 | 降级到FallbackLLM |
| Browser Pool  | 连续3次启动失败 | 暂停分配+重建池       |
| Cache Backend | 连续5次读写失败 | 降级到内存缓存        |
| Plugin System | 插件连续3次异常 | 禁用插件+通知        |

### 29.5 审计日志与OTel互补机制

> 审计日志与OpenTelemetry形成互补，分别提供业务决策审计和技术链路追踪能力。

| 维度 | OpenTelemetry | 审计日志 |
| --- | --- | --- |
| **视角** | 技术视角 | 合规视角 |
| **关注点** | 性能、延迟、错误 | AI决策、人工介入、敏感操作 |
| **数据类型** | 指标、链路、日志 | 不可篡改操作轨迹 |
| **用途** | 问题定位、性能优化 | 合规审计、AI决策可解释性 |

**互补机制**：

```
执行流程
    │
    ├── OpenTelemetry追踪
    │   ├── 技术链路：LLM调用耗时、网络请求延迟、执行步骤耗时
    │   ├── 性能分析：瓶颈识别、资源消耗、异常定位

    │
    ├── 审计日志记录
    │   ├── 业务决策：AI选择哪个动作、为什么选择、决策依据
    │   ├── 人工介入：何时介入、介入原因、介入结果
    │   ├── 敏感操作：访问敏感数据、执行敏感动作、审批记录
    │   └── 不可篡改：写入时序数据库、访问审计、合规报告
    │
    └── 互补应用
        ├── 问题诊断：OTel定位技术瓶颈 + 审计日志追溯决策链
        ├── 合规审计：审计日志提供合规证据 + OTel提供技术支撑
        └── AI可解释性：审计日志记录AI决策过程 + OTel追踪执行轨迹
```

> **设计原则**：OTel提供技术视角（性能、延迟、错误），审计日志提供合规视角（AI决策、人工介入、敏感操作）。两者互补，形成完整的可观测性和可审计性体系。

## 38. 关键设计决策

| 决策          | 选择                            | 原因                                        |
| ----------- | ----------------------------- | ----------------------------------------- |
| **执行方式**    | 进程内直调Playwright SDK           | 消除MCP RPC开销，生产环境稳定                        |
| **Agent循环** | 自研PlannerAgent（参考browser-use） | 完全可控，多模型适配                                |
| **感知模式**    | A11ySnapshot优先，DOM/视觉降级，混合模式独立可选 | A11ySnapshot低token，DOM可录制代码，视觉覆盖Canvas等场景 |
| **LLM提供商**  | OpenAI / Qwen / Ollama 多选     | 用户自由选择，无限制                                |
| **意图路由**    | 多模型意图路由（locate按感知模式路由：DOM→LLM/视觉→VL）   | 按意图+感知模式选最优模型，平衡精度与成本                          |
| **循环检测**    | LoopDetector跟踪历史              | 防止死循环                                     |
| **代码录制**    | 执行同步录制                        | 执行完即可回放                                   |
| **缓存策略**    | 三层缓存（Plan/Locate/Feature）     | 首次正常调用，后续命中提速3-10倍，降低token                |
| **定位策略**    | 两阶段深度定位（区域→元素）                | 减少token消耗，提高复杂页面精度                        |
| **自愈策略**    | 四层降级（选择器→DOM→视觉→AI修复）         | 确定性优先，AI兜底，人工审批                           |
| **知识管理**    | 分级沉淀（需求级/产品级/经验级）             | 经验自动积累，跨用例复用                              |
| **消息管理**    | 自动压缩（超阈值压缩历史为摘要）              | 控制长流程token消耗                              |
| **结果评估**    | 独立JudgeAgent                  | 避免自我评估偏差，提高结果可信度                          |
| **守卫系统**    | Watchdog事件驱动                  | 弹窗/崩溃/CAPTCHA自动处理，生产可靠                    |
| **安全保护**    | 敏感数据脱敏 + 域名过滤                 | 防止凭证泄露和越权访问                               |
| **可靠性**     | FallbackLLM + ActionTimeout   | 主模型故障降级，动作挂起保护                            |
| **断言体系**    | 四层断言（结构/视觉/语义/行为）             | 覆盖从DOM到AI的完整验证能力                          |
| **能力封装**    | SkillRegistry                 | 可复用测试能力单元，Agent按需调用                       |
| **错误处理**    | 熔断器+舱壁隔离+优雅降级                 | 系统级容错，防止级联故障                              |
| **可观测性**    | OpenTelemetry三支柱              | 全链路可观测，快速定位问题                             |
| **成本治理**    | 意图路由+缓存+Flash Mode多维优化        | 多维优化LLM调用成本                               |
| **API设计**   | 渐进式四层SDK（NL→原子→原生→平台）         | 不同用户群体，不同控制粒度                             |
| **编排调度**    | DAG编排引擎                       | 支持复杂依赖关系和并行策略                             |
| **部署模式**    | Sandbox容器化部署                  | 环境隔离，资源限制，安全可控                            |
| **降级命名**    | D1-D4降级链与R1-R3运行模式区分命名空间   | 避免概念混淆，D系列为系统降级策略，R系列为运行模式          |
| **感知-路由两级** | 感知模式决定"怎么看页面"，意图路由决定"用哪个模型" | 两级解耦，感知模式约束意图路由，避免跨级决策               |
| **缓存回退闭环** | 缓存命中执行失败→清除缓存→降级AI完整调用   | 缓存不是银弹，执行失败时必须回退到完整AI调用              |
| **检查点恢复**   | CrashWatchdog恢复到最近检查点（URL+StorageState） | 比恢复到崩溃前页面更可靠，从检查点重新执行             |
| **2FA/OTP**    | CaptchaWatchdog支持pyotp自动处理     | 自动处理2FA验证码，无法自动处理时通知人工              |
| **全局重试上限** | 全局10次重试上限，含LLM+自愈+JudgeAgent  | 防止重试叠加导致执行时间爆炸                          |
| **Spec-Code对账** | Spec变更自动触发代码对账             | 确保测试代码始终反映最新需求                          |

### 39.1 CodeRecorder 录制机制

> CodeRecorder在ExecutorAgent执行每个动作时同步录制Playwright代码，支持回放和脚本化运行。

**触发时机**：ExecutorAgent执行每个动作时顺便记录

**参数配置**：可通过参数配置是否需要录制

| 参数 | 默认值 | 说明 |
| --- | --- | --- |
| `record_enabled` | true | 是否启用录制 |
| `record_mode` | semantic | 录制模式：semantic（语义化）/ raw（原始） |
| `record_output` | file | 输出方式：file（文件）/ memory（内存） |

**录制能力矩阵**：

| 执行端 | 录制支持 | 说明 |
| --- | --- | --- |
| **Web端** | ✅ 支持 | 录制Playwright代码，语义化选择器优先 |
| **全视觉模式** | ❌ 不支持 | 无法生成稳定代码，仅记录坐标+OCR序列 |
| **其他端（App等）** | ❌ 不支持 | 暂不支持录制，后续扩展 |

**录制流程**：

```
ExecutorAgent执行动作
    │
    ├── CodeRecorder拦截
    │   ├── 检查record_enabled配置
    │   ├── 检查执行模式（DOM_MODE才录制）
    │   └── 检查执行端（Web端才录制）
    │
    ├── 动作→代码映射
    │   ├── CLICK → page.click(selector)
    │   ├── TYPE → page.fill(selector, text)
    │   ├── NAVIGATE → page.goto(url)
    │   ├── SELECT → page.select_option(selector, value)
    │   ├── SCROLL → page.mouse.wheel(delta)
    │   ├── WAIT → page.wait_for_selector(selector)
    │   └── ASSERT → expect(locator).to_xxx()
    │
    ├── 选择器生成策略
    │   ├── 优先级1: getByTestId（最稳定）
    │   ├── 优先级2: getByRole + getByLabel
    │   ├── 优先级3: CSS选择器
    │   └── 优先级4: XPath（兜底）
    │
    └── 代码输出
        ├── 语义化代码片段
        ├── 完整测试脚本组装
        └── 导出为.py文件
```

> **设计原则**：录制代码优先使用语义化选择器（getByTestId/getByRole），确保代码可读性和稳定性。视觉模式不录制代码，但记录坐标+OCR序列供回放使用。

> **设计声明**：本框架采用六层架构设计，支持多种LLM提供商，用户可自由选择。所有AI修复操作默认需人工审批。缓存命中时自动跳过AI调用，未命中时降级为完整AI调用；缓存执行失败时清除缓存并降级为AI完整调用。Watchdog守卫系统（含UI样式异常检测、2FA/OTP自动处理、检查点恢复）确保生产环境可靠性。系统级容错通过熔断器+舱壁隔离+优雅降级（D1-D4与R1-R3区分命名空间）保障。全链路可观测性基于OpenTelemetry三支柱，审计日志确保合规可审计性。代码录制机制同步生成语义化Playwright代码，视觉模式通过坐标+OCR序列回放弥补无法录制代码的缺陷。InputPrimitives跨平台输入原语统一多端动作空间。业务RAG使Agent具备业务理解能力，受上下文窗口预算约束。检查点机制应对长链路成功率衰减。感知-路由两级决策：感知模式决定"怎么看页面"，意图路由决定"用哪个模型"，感知模式约束意图路由。视觉模式敏感数据保护（截图区域脱敏/OCR检测脱敏/历史截图脱敏）是安全底线。Skill仅保留高层业务封装（业务流程/验证断言/数据操作），底层操作由原子ACTION覆盖。全局重试上限10次（含LLM+自愈+JudgeAgent），防止重试叠加。Spec-Code对账确保测试代码始终反映最新需求。成本治理通过意图路由+缓存+Flash Mode+Capability多维优化。API设计遵循渐进式原则，从自然语言到原生API四层覆盖。Playwright通过进程内直调SDK集成，MCP仅限本地开发辅助。知识图谱、A/B测试、多租户、DAG编排等高级能力分阶段实施，避免过度设计。
