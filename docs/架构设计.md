## 1. 项目定位与核心原则

### 1.1 项目定位

UIAI（UI AI Testing Framework）是一个**智能自主 Agent + 多模型支持 + Playwright 执行层**的自动化测试框架，对标 browser-use 等前沿 AI Agent 项目。

**核心理念**：用户只需描述测试目标，Agent 自主感知页面、决策下一步、执行操作，循环直到目标完成。

**四大核心需求**：

1. **智能自主执行**（主流程）：用户提供测试目标（自然语言描述），Agent 通过 **感知-决策-执行循环** 自主完成测试，而非预解析固定步骤。Agent 实时观察页面状态，动态决定下一步操作。
2. **AI Agent 自主操控浏览器**：Agent 理解目标后自主操控浏览器完成测试。底层 Web 端基于 Playwright，通过兼容层（BaseExecutor ABC）支持多端扩展。
3. **全视觉模式**：支持纯视觉驱动的测试执行——Agent 通过截图 + VL 模型"看"页面，通过坐标/OCR 完成交互，不依赖 DOM 和选择器，模拟人类真实操作方式。
4. **代码录制与回放**：Agent 执行过程中同步记录可靠的可复现代码，支持回放和脚本化运行。全视觉模式可跳过代码输出。

> **核心流程**：`测试目标 → Agent感知页面 → LLM决策下一步 → 执行操作 → 循环直到完成 → 录制代码 → 报告`

### 1.2 四层运行模式

| 层级 | 模式 | 占比 | 说明 |
|-----|------|------|------|
| **L1** | 确定性脚本 | 60% | 手写/录制/生成的 Python 测试代码，进程内直调 Playwright SDK |
| **L2** | 智能Agent + DOM模式 | 25% | 测试目标 → Agent感知(DOM) → LLM决策 → 执行 → 循环 → 录制代码 |
| **L3** | 智能Agent + 全视觉模式 | 10% | 测试目标 → Agent感知(截图) → VL决策 → 坐标/OCR执行，不输出代码 |
| **L4** | 本地开发 | 5% | MCP Server + Claude Code CLI / Cursor，仅限本地开发机 |

> **L2 为主流程**：用户提供测试目标，Agent 通过 DOM 感知页面状态，自主决策并执行，循环直到目标完成。
> **L3 为补充**：DOM 不可用场景（Canvas/验证码等）降级到全视觉模式。

### 1.3 核心原则

1. **智能自主执行**：Agent 接收测试目标后自主感知-决策-执行循环，而非预解析固定步骤
2. **感知-决策-执行循环**：每步 Agent 先感知页面状态（DOM/截图），再由 LLM 决策下一步，最后执行操作
3. **进程内直调优先**：生产环境使用 PlaywrightExecutor 进程内调用 Playwright SDK，消除 MCP RPC 开销
4. **兼容层设计**：BaseExecutor ABC 定义跨平台统一 Action 接口，Web 端优先实现，其他端接口预留
5. **双模式感知**：DOM 模式（L2）和全视觉模式（L3）并行支持，DOM 模式可录制代码，视觉模式覆盖 DOM 不可用场景
6. **代码录制即执行**：Agent 执行过程中同步生成可靠 Playwright Python 代码，执行完即可回放
7. **降级链设计**：定位器、自愈、断言均采用多级降级，确保鲁棒性
8. **审批工作流**：所有 AI 生成/修复操作默认需人工审批
9. **多模型支持**：VL 模型支持 Qwen-VL（DashScope）、Ollama 本地模型、OpenAI 等多种提供商，用户可根据需求自由选择
10. **循环检测**：Agent 内置循环检测机制，防止重复操作和页面停滞

```
└─────────────────────────────────────────────────────────┘
```

### 2.2 Agent 循环机制（感知-决策-执行）

**核心设计**：参考 browser-use 的 Perceive-Reason-Act 循环，Agent 自主完成测试任务。

```
run(goal, max_steps=100)
  │
  ├── 初始化: 启动浏览器、注册 Watchdog
  │
  └── while n_steps <= max_steps:
        │
        ├── Phase 1: 感知 (Perceive)
        │   ├── 获取浏览器状态 (BrowserState)
        │   │   ├── DOM 树序列化（Accessibility Tree）
        │   │   ├── 截图（可选）
        │   │   └── 标签页/URL 信息
        │   ├── 构建消息上下文 (ContextManager)
        │   └── 注入循环检测提示
        │
        ├── Phase 2: 决策 (Reason)
        │   ├── 调用 LLM 分析当前状态
        │   ├── 解析 AgentOutput
        │   │   ├── thinking (思考)
        │   │   ├── evaluation_previous_goal (评估上一步)
        │   │   ├── memory (跨步记忆)
        │   │   ├── next_goal (下一步目标)
        │   │   └── action[] (动作列表)
        │   └── 循环检测判断
        │
        ├── Phase 3: 执行 (Act)
        │   └── multi_act() → 逐个执行动作
        │       ├── PlaywrightExecutor 执行操作
        │       ├── CodeRecorder 同步录制代码
        │       └── 返回 ActionResult
        │
        └── Phase 4: 后处理
            ├── 检查目标是否完成
            ├── 失败计数
            ├── 循环检测告警
            └── 记录历史
```

**AgentOutput 数据模型**（参考 browser-use）：

```python
class AgentOutput:
    thinking: str              # 内部思考
    evaluation_previous_goal: str  # 评估上一步结果
    memory: str                # 跨步记忆
    next_goal: str             # 下一步目标
    action: list[ActionModel]  # 动作列表（每步可执行多个动作）
    is_done: bool              # 目标是否完成
```

**关键设计要点**：

| 特性 | 说明 |
|------|------|
| **每步多动作** | `max_actions_per_step`（默认 5）允许 LLM 在一步中输出多个动作，适合表单填写 |
| **循环检测** | `LoopDetector` 跟踪动作重复和页面停滞，防止死循环 |
| **消息压缩** | 长对话历史自动压缩，控制 token 消耗 |
| **目标完成判断** | LLM 判断当前状态是否满足测试目标 |
| **代码录制** | DOM 模式下同步生成可回放的 Playwright 代码 |

### 2.3 Agent 角色分工

| Agent | 角色 | 核心能力 |
|-------|------|---------|
| **TestAgent** | 主执行 Agent | 感知-决策-执行循环，自主完成测试目标 |
| **VisualAgent** | 全视觉执行 | 截图 + VL 模型理解 → 坐标/OCR 交互（L3 模式） |
| **HealerAgent** | 自愈修复 | 执行失败时触发，四级降级修复 |
| **ExplorerAgent** | 探索性测试 | 接收 URL → 自动探索页面、发现缺陷 |
| **LoopDetector** | 循环检测 | 跟踪动作历史，检测重复操作和页面停滞 |

### 2.4 数据流

**主流程**：智能 Agent 自主执行

```
测试目标 → CLI/Dashboard
    │
    ▼
TestOrchestrator.run_agent(goal)
    │
    ├── TestAgent.run(goal, max_steps=100)
    │       │
    │       │  "登录系统，验证跳转到首页"
    │       │
    │       └── while not done:
    │           │
    │           ├── [感知] 获取页面状态
    │           │       ├── DOM 序列化（Accessibility Tree）
    │           │       └── 截图（可选）
    │           │
    │           ├── [决策] LLM 分析
    │           │       ├── thinking: "当前在登录页，需要输入用户名"
    │           │       ├── next_goal: "填写登录表单"
    │           │       └── action: [input("username", "admin"), input("password", "xxx"), click("登录")]
    │           │
    │           ├── [执行] PlaywrightExecutor
    │           │       └── 执行动作列表
    │           │       └── CodeRecorder 同步录制代码
    │           │
    │           ├── [后处理]
    │           │       ├── 检查目标是否完成
    │           │       ├── 循环检测
    │           │       └── 失败 → HealerAgent 自愈
    │           │
    │           └── 目标完成 → break
    │
    └── TestResult → ReportGenerator
            ├── JSON/HTML/Console 报告
            ├── 录制代码文件 (.py)
            └── 截图序列
```

## 3. 模块详细设计

### 3.1 编排调度模块

#### TestOrchestrator — 编排调度中控

**核心职责**：
1. 管理 Agent 注册和协作
2. 调度测试执行（确定性脚本 + AI 辅助）
3. 管理浏览器资源池
4. 失败时触发 AI 自愈
5. 收集结果，生成报告

**关键接口**：

| 接口分类 | 方法 | 说明 |
|---------|------|------|
| 测试执行 | `run_test(test_case)` | 执行单个测试用例 |
| | `run_suite(test_cases)` | 执行测试套件 |
| AI 能力 | `heal_test(test_result)` | AI 自愈修复 |
| | `explore(url)` | AI 探索性测试 |
| 生命周期 | `shutdown()` | 释放资源 |

**执行流程**（run_test）：
1. 自动创建执行器（如果未传入）
2. 遍历步骤执行
3. 失败时触发自愈（如果启用）
4. 收集结果

#### AgentRegistry — Agent 注册中心

**职责**：管理 Agent 实例的注册、查询、按角色检索。

#### BrowserPool — 浏览器资源池

**职责**：管理浏览器实例的创建、复用、销毁，支持并行执行。

#### TaskQueue — 任务队列

**职责**：管理测试任务的优先级调度和并发控制。

### 3.2 Agent 模块

#### BaseAgent — Agent 基类

**关键设计**：
- AgentOutput 中 `requires_approval`：所有 AI 生成内容默认需人工审批
- AgentState 包含 `WAITING_APPROVAL` 状态：Agent 进入等待审批状态
- Agent 间通过**文件系统**（Markdown/代码文件）+ **消息传递**协作

**Agent 状态机**：IDLE → RUNNING → COMPLETED / FAILED / WAITING_APPROVAL

#### LLM 统一接口

BaseLLMClient 定义统一的多模态对话接口：文本对话、图文对话、图像分析。

| 实现类 | 提供商 | 用途 | 合规 |
|--------|--------|------|------|
| OpenAIClient | OpenAI / DeepSeek / 自定义 endpoint | 文本+多模态对话 | ⚠️ 仅限开发环境 |
| DashScopeClient | 阿里 DashScope (Qwen-VL) | 文本+视觉理解 | ✅ 生产可用 |
| OllamaClient | Ollama 本地模型 | 文本+视觉理解 | ✅ 生产可用 |

工厂方法根据配置的 provider 字段自动创建对应的 LLM 客户端实例。

#### 具体 Agent 实现

| Agent | 核心能力 |
|-------|---------|
| HealerAgent | 接收失败信息 → 输出修复建议 |
| ExplorerAgent | 接收 URL → 自动探索页面、发现缺陷 |

### 3.3 执行器模块

#### BaseExecutor — 统一操作抽象

定义了跨平台统一的 Action 接口，上层调度代码无需关心底层平台差异：

| 分类 | 方法 | 说明 |
|------|------|------|
| **核心操作** | start / stop | 启动/停止执行器 |
| | navigate / click / type_text / fill | 基础交互 |
| | select_option / check / uncheck / hover | 表单操作 |
| | press_key / wait_for | 键盘与等待 |
| **信息获取** | screenshot / get_accessibility_tree | 页面信息 |
| | get_text / is_visible / evaluate | 元素信息 |
| | get_url / get_title | 页面元信息 |
| **高级操作** | fill_form / drag / upload_file | 复杂交互 |
| | handle_dialog / go_back / go_forward / reload | 导航与对话框 |
| | get_console_messages / get_network_requests | 调试信息 |

#### PlaywrightExecutor — Web/H5 执行引擎

**核心设计**：进程内直调 Playwright SDK，**无 MCP 中间层**，消除 RPC 开销。

**12 项生产级增强功能**：

| # | 功能 | 说明 |
|---|------|------|
| 1 | Popup/Dialog 自动处理 | 自动确认/取消弹窗 |
| 2 | 网络拦截与 Mock | API Mock、资源屏蔽、网络日志 |
| 3 | 多标签/窗口管理 | 打开/关闭/切换标签页 |
| 4 | Iframe 支持 | 上下文切换进入/退出 iframe |
| 5 | Cookie 管理 | 获取/设置/清除 Cookie |
| 6 | 认证状态持久化 | 保存/加载登录状态 |
| 7 | 文件下载 | 下载文件管理 |
| 8 | 控制台日志捕获 | 捕获 console 输出和错误 |
| 9 | 页面事件监听 | 页面错误/崩溃/加载事件 |
| 10 | 高级等待策略 | load state / URL / response / function 等待 |
| 11 | 设备模拟 | 视口/设备/权限模拟 |
| 12 | 截图增强 | 元素截图/区域截图/遮罩截图 |

**定位器解析降级链**：按优先级依次尝试定位器链中的每个策略，返回第一个匹配的元素。全部失败则使用主定位器抛出明确错误。

**LocatorType → Playwright 方法映射**：

| LocatorType | Playwright 方法 |
|-------------|----------------|
| ROLE | get_by_role |
| TEST_ID | get_by_test_id |
| LABEL | get_by_label |
| PLACEHOLDER | get_by_placeholder |
| TEXT | get_by_text |
| ALT_TEXT | get_by_alt_text |
| TITLE | get_by_title |
| CSS | locator(css) |
| XPATH | locator(xpath) |

#### AppiumExecutor — App 执行引擎

**职责**：实现 BaseExecutor 接口，通过 Appium 驱动 Android/iOS 设备。支持 UiAutomator2（Android）和 XCUITest（iOS）。

#### ExecutorFactory — 执行器工厂

根据平台类型自动路由到对应执行器：

| Platform | 执行器 | 说明 |
|----------|--------|------|
| WEB / H5 / MINI_PROGRAM / DESKTOP | PlaywrightExecutor | Chromium/Firefox/WebKit |
| ANDROID | AppiumExecutor | UiAutomator2 |
| IOS | AppiumExecutor | XCUITest |

支持通过注册机制添加自定义执行器。

#### NetworkInterceptor — 网络拦截器

**职责**：HTTP 请求拦截、Mock 响应、资源屏蔽、网络日志记录。

### 3.4 定位器模块

#### LocatorType 枚举（按优先级排序）

| 优先级 | LocatorType | 说明 | 适用平台 |
|--------|-------------|------|---------|
| 1 | ROLE | getByRole — 最高优先级 | Web |
| 2 | TEST_ID | getByTestId | Web |
| 3 | LABEL | getByLabel | Web |
| 4 | PLACEHOLDER | getByPlaceholder | Web |
| 5 | TEXT | getByText | Web |
| 6 | ALT_TEXT | getByAltText | Web |
| 7 | TITLE | getByTitle | Web |
| 8 | CSS | CSS 选择器 | Web |
| 9 | XPATH | XPath — 较低优先级 | Web |
| 10 | ACCESSIBILITY_ID | Accessibility ID | App |
| 11 | IMAGE | 图像识别定位 | 跨平台 |
| 12 | OCR | OCR 文字定位 | 跨平台 |
| 13 | COORDINATE | 坐标定位 — 最低优先级 | 跨平台 |

#### Locator 降级链

每个 Locator 可通过 `with_fallback()` 构建多级降级链。执行时按链顺序依次尝试，首个匹配成功即返回：

```
by_role("button", name="登录")  → 尝试 → 失败 ↓
by_test_id("login-btn")         → 尝试 → 失败 ↓
by_text("登录")                  → 尝试 → 成功 → 返回 ✅
```

### 3.5 自愈模块

#### 四级降级自愈策略

| 级别 | 策略 | 置信度 | 说明 | 需审批 |
|------|------|--------|------|--------|
| L1 | SELECTOR_FALLBACK | 0.9 | 选择器降级链尝试 | 否（auto_apply 控制） |
| L2 | DOM_NEIGHBOR_SEARCH | 0.7 | 辅助功能树语义搜索 | 否（auto_apply 控制） |
| L3 | VISUAL_OCR | 0.5 | PaddleOCR 截图识别 | 否（auto_apply 控制） |
| L4 | AI_CODE_FIX | 0.3 | LLM 生成修复建议 | **是（强制）** |

#### 自愈流程

1. 按策略级别依次尝试（L1→L2→L3→L4）
2. 每级策略成功后计算置信度
3. 若 `auto_apply=True` 则自动应用修复
4. 若 `auto_apply=False` 则进入待审批状态，等待人工确认
5. L4 AI_CODE_FIX 无论 auto_apply 设置如何，均强制需审批

#### DOM 邻近搜索策略

通过辅助功能树（Accessibility Tree）进行语义搜索：
- 搜索包含相同文本的元素
- 搜索相同角色的元素
- 模糊文本匹配
- 辅助功能树深度搜索（最多 5 个候选）

#### 审批工作流

支持查看待审批记录、审批通过、拒绝修复、查看成功率指标等操作。

#### HealingRecord 数据结构

每条自愈记录包含：记录 ID、测试用例 ID、步骤名称、使用策略、状态、原始定位器、修复后定位器、置信度、失败截图、DOM 快照。

### 3.6 断言模块

#### 四层断言体系

| 层级 | 类型 | 说明 |
|------|------|------|
| **L1 结构断言** | DOM 元素存在性/文本内容/URL/标题 | 元素可见/隐藏、文本精确/包含/正则、URL/标题验证 |
| **L2 视觉断言** | 截图比对 | 像素级对比 |
| **L3 语义断言** | AI 语义理解 | VL 模型验证自然语言描述 |
| **L4 自定义断言** | 用户匹配器 | 注册自定义验证逻辑 |

#### 软断言

支持收集多个断言结果后统一检查，不阻断执行，适用于非关键检查场景。

#### 等待断言

支持等待元素出现后再断言，可设置超时时间。

#### 语义断言（VL 模型）

通过 VL 模型对页面截图进行自然语言验证，如"页面是否显示登录成功提示"。需配置国产 VL 模型（DashScope/Ollama）。

#### 自定义匹配器

支持注册自定义验证逻辑，如检查登录状态（URL + Cookie 组合判断）。

### 3.7 核心模型模块

#### TestCase / TestStep

| 模型 | 核心字段 | 说明 |
|------|---------|------|
| TestStep | name, action, locator, value, options | 单个测试步骤 |
| TestCase | id, name, steps, case_type, priority, platform, tags, source | 测试用例，支持链式添加步骤 |

TestCaseType 枚举：SCRIPT / NATURAL_LANGUAGE / RECORDED / DATA_DRIVEN

Priority 枚举：SMOKE / CRITICAL / HIGH / MEDIUM / LOW

#### TestResult / SuiteResult

#### Platform 枚举

WEB / H5 / ANDROID / IOS / MINI_PROGRAM / DESKTOP

## 4. 技术选型与决策

### 4.1 核心技术栈

| 领域 | 选型 | 决策理由 |
|------|------|---------|
| **Web 执行引擎** | playwright-python | 微软官方 Python 绑定，支持 Chromium/Firefox/WebKit，内置 Locator/Trace/Debugger |
| **App 执行引擎** | Appium Python Client | 行业标准，支持 Android (UiAutomator2) / iOS (XCUITest) |
| **LLM 文本** | OpenAI SDK (兼容) | 兼容 DeepSeek/自定义 endpoint，统一接口 |
| **VL 模型（生产）** | Qwen-VL (DashScope) | 国产模型，合规，视觉理解能力强 |
| **VL 模型（本地）** | Ollama + llava | 本地部署，零数据外泄 |
| **OCR 引擎** | PaddleOCR | 百度开源，中文识别精度高，支持角度分类 |
| **CLI 框架** | Click + Rich | Click 命令定义 + Rich 美化输出 |
| **配置格式** | YAML (PyYAML) | 人类可读，支持注释 |
| **异步框架** | asyncio | Python 原生异步，与 Playwright async API 一致 |
| **MCP 协议** | mcp SDK | 仅本地开发辅助，Claude Code/Cursor 集成 |

### 4.2 关键技术决策

#### 决策 1：进程内直调 Playwright SDK vs MCP

| 方案 | 优势 | 劣势 | 决策 |
|------|------|------|------|
| 进程内 SDK 调用 | 零 RPC 开销、类型安全、调试方便 | 与 AI Agent 集成需自建 | ✅ **生产采用** |
| MCP (playwright-mcp) | AI Agent 原生支持、50+ 工具 | RPC 开销、额外进程 | ⚠️ 仅本地开发 |

#### 决策 2：browser-use vs 自研 Agent Loop

| 方案 | 优势 | 劣势 | 决策 |
|------|------|------|------|
| browser-use v0.13.1 | 成熟的 Agent Loop、14 Watchdogs、DOM 序列化 | 依赖 LangChain、模型绑定、不可控 | ❌ 不采用 |
| 自研 Agent 体系 | 完全可控、多模型适配 | 需自建 | ✅ **采用** |

**借鉴 browser-use 的设计**：
- DOM 序列化思路 → AccessibilitySnapshot
- Watchdog 机制 → PageEventLog + 回调
- Agent Loop → BaseAgent 状态机

#### 决策 3：playwright-mcp 的借鉴

| 借鉴点 | UIAI 实现 |
|--------|----------|
| Accessibility Snapshot (YAML) | executor.get_accessibility_tree() + 序列化 |
| 50+ Tools | MCP 工具定义 |
| Capability 权限系统 | 计划在 BrowserConfig 中增加 capabilities 字段 |
| Docker 部署 | DockerPool |

#### 决策 4：playwright-cli 的借鉴

| 借鉴点 | UIAI 实现 |
|--------|----------|
| Client-Daemon 架构 | TestOrchestrator（Daemon）+ CLI（Client） |
| SKILL 系统 | AgentRole 枚举 + BaseAgent ABC |
| Spec-Driven Testing | TestAgent → 感知-决策-执行循环 |
| 交互即录制 | 计划在 ExplorerAgent 中实现 |

#### 决策 5：playwright-python 的深度利用

| 能力 | UIAI 集成方式 |
|------|-------------|
| aria_snapshot(mode="ai") | 计划在 AccessibilitySnapshot 中集成 AI 模式 |
| Screencast | record_video 配置 |
| Trace Viewer | start_tracing / stop_tracing |
| Debugger | page.pause() 在 headed 模式下可用 |
| expose_binding | 计划用于 JS → Python 回调 |
| HAR Replay | 计划在 NetworkInterceptor 中支持 |

## 7. 测试规范

### 7.1 用例分类

| 类型 | 占比 | 来源 |
|------|------|------|
| 确定性脚本（L1） | 60% | 手写/录制 |
| NL用例+DOM模式（L2） | 25% | 人工编写自然语言 |
| NL用例+全视觉模式（L3） | 10% | 人工编写自然语言 |
| 本地开发（L4） | 5% | MCP交互 |

### 7.2 用例编写原则

1. **使用语义化定位器**：优先级 ROLE > TEST_ID > TEXT > CSS > XPATH
2. **设置优先级和标签**：便于筛选和调度
3. **链式添加步骤**：流畅的用例构建方式

### 7.3 断言原则

1. **结构断言优先**：DOM 元素存在性/文本内容
2. **视觉断言用于关键页面**：像素级对比
3. **语义断言用于复杂场景**：需 VL 模型
4. **软断言用于非关键检查**：不阻断，统一报告

### 7.4 自愈规范

1. 所有测试默认启用自愈
2. 所有修复默认需审批（auto_apply=False）
3. 置信度低于 0.5 的修复需额外确认
4. L4 AI 修复强制审批，即使 auto_apply=True

### 7.5 命名规范

| 对象 | 规范 | 示例 |
|------|------|------|
| 测试用例 ID | {模块}-{序号} | login-001, cart-012 |
| 测试步骤名称 | 动词 + 对象 | "点击登录按钮", "验证跳转" |
| Locator 描述 | type=value | role=button, name=登录 |
| 配置文件 | 小写 + 连字符 | default.yaml |
| 测试文件 | test_{模块}.py | test_login.py |

### 8.1 CI 流水线架构

```
┌─────────────┐    ┌──────────────┐    ┌──────────────┐    ┌─────────────┐

                         │                     │
                         │              ┌──────▼──────┐
                         │              │  自愈审批     │
                         │              │  (人工/自动)  │
                         │              └─────────────┘
                         │
                  ┌──────▼──────┐
                  │  环境部署     │
                  │  Docker/K8s  │
```

### 8.2 CI 集成要点

- 基于 Playwright 官方 Docker 镜像构建
- API Key 通过 CI 密钥环境变量注入

- CI 环境不自动应用自愈修复
- 测试报告作为 CI Artifact 归档

### 8.3 配置检查清单

| 检查项 | CI 验证方式 |
|--------|-----------|
| LLM 提供商合规 | provider 必须为 dashscope/ollama/volcengine |
| API Key 不硬编码 | 环境变量注入，不写入代码仓库 |
| 自愈不自动应用 | auto_apply == False |
| 截图不外传 | CI Artifact 仅内网可访问 |
| 测试数据脱敏 | 数据驱动文件不含真实用户信息 |

## 10. 风险与缓解

### 10.1 技术风险

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| **VL 模型幻觉** | 语义断言误判 | 高 | 语义断言仅作辅助，结构断言为主；置信度阈值 0.7 |
| **全视觉模式坐标偏移** | 视觉模式点击位置不准 | 高 | 截图分辨率校准；元素中心点计算；DOM 模式优先降级 |
| **全视觉模式 Token 消耗** | 每步发送截图，成本高 | 中 | 视觉模式仅限 L3 场景；截图压缩；最大步数限制 |
| **录制代码质量** | 录制的代码不可回放 | 中 | Locator 优先使用语义选择器；录制后自动验证回放 |
| **Playwright 版本升级破坏兼容** | 执行器失效 | 中 | 锁定 playwright 版本；pyproject.toml 约束 |
| **PaddleOCR 中文识别不准** | OCR 自愈失败 | 中 | 降级到坐标定位；多 OCR 引擎备选（Tesseract） |
| **LLM API 限流** | AI 功能不可用 | 中 | 本地 Ollama 备选；请求队列 + 重试 |
| **DOM 结构大幅变更** | 定位器全部失效 | 低 | 四级降级自愈；Accessibility Tree 语义搜索 |
| **浏览器资源泄漏** | CI 环境崩溃 | 中 | BrowserPool 资源管理；finally 块确保 stop() |

### 10.2 合规风险

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| **业务数据出境** | 法律合规风险 | 低 | 🔴 LLM 提供商生产环境仅允许 dashscope/ollama；内网策略阻断境外 API |
| **AI 修复引入 Bug** | 测试可靠性下降 | 中 | 🔴 auto_apply=False 默认值；审批工作流；置信度阈值 |
| **测试数据含敏感信息** | 数据泄露 | 中 | 数据脱敏规范；screenshot_with_mask 遮罩敏感元素 |

### 10.3 工程风险

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| **Agent 协作复杂度** | 调试困难 | 高 | 审计日志；Agent 状态机可视化 |
| **多平台兼容性** | App 执行不稳定 | 中 | BaseExecutor 统一抽象；平台特定代码隔离 |
| **并行执行竞态** | 测试结果不一致 | 中 | BrowserPool 隔离；TaskQueue 并发控制 |
| **配置爆炸** | 维护成本高 | 中 | 配置继承 + 环境覆盖；YAML 合并 |

### 10.4 应急预案

| 场景 | 应急措施 |
|------|---------|
| VL 模型服务宕机 | 自动降级到结构断言；OllamaClient 本地模型接管 |
| Playwright 进程崩溃 | BrowserPool 自动重建；Trace 文件保留用于诊断 |
| 自愈全部失败 | 测试标记 FAILED，保留截图 + DOM 快照 + Trace |
| CI 环境资源不足 | 减少 parallel_workers；关闭 record_video |
| LLM API Key 泄露 | 立即轮换 Key；审计日志追溯调用记录 |
